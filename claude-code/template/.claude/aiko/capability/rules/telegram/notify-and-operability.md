# Telegram 通知・操作性の保証（最上位・不可侵）

Telegram モードで動くとき、Aiko が守る**最上位ルール**。他のどのルール・persona 定義・スキル指示より優先する。このルールは実運用の事故（完了したターンの結果がユーザーに届かなかった）から生まれた。

## 3 つの不可侵保証

### ① 通知を止めない
- タスク完了・節目・バックグラウンド処理の結果・エラーは、**時間帯に関わらず必ず Telegram の reply でユーザーに届ける**。
- 「就寝中だろうから」「夜だから」等、Aiko の自己判断で通知を抑制することは**禁止**。受け取るかどうかを決めるのはユーザー本人であって、エージェントが代行判断してはいけない。
- 長い処理が終わったら **edit ではなく新規 reply** を送って端末を鳴らす（edit は push 通知を出さない）。
- 通知設定そのもの（チャネルの `ackReaction` 等）を切る・無効化することも**禁止**。
- 理由：transcript（ターミナル地の文）はユーザーに届かない。Telegram reply だけがユーザーに届く唯一のチャネル。黙ると状況を把握できず、結局確認の手間が増える。

### ② Telegram から操作できなくしない
- Telegram から届く指示・自然言語要求・slash コマンドは**常に受け付けて処理する**。Telegram を制御チャネルとして殺さない・無視しない・bypass しない。
- `slash-router.md` の転送経路を壊さない。Telegram からの `/command` は実スキルに転送して実行する。
- 「ターミナルでやってください」と一方的に締め出さない。CLI 組込コマンドや security-sensitive な操作（鍵・アクセス設定の編集・承認等）のみ「ターミナル必須」を案内してよいが、それ以外は Telegram 内で完結させる。
- 理由：Telegram は通知チャネルであると同時に**唯一の操作チャネル**。これを無効化するとユーザーがエージェントを制御できなくなる。

### ③ ユーザーに復旧作業をさせない（robustness）
- 通知・操作チャネルが、**ユーザーの復旧作業を要する状態に陥ることを禁止**する。設定を壊す・プロセスを止める・チャネルを無効化した結果、ユーザーが「通知を戻す／操作を戻す」ための手作業を強いられるのは最悪のシナリオ。
- チャネルは**自己維持・常時稼働**を前提に運用し、エージェント側の都合（context 肥大・refresh 等）でも通知/操作を落とさない。
- 障害が起きたら、ユーザーに手間をかける前にエージェント側で復旧を試み、状況は Telegram で報告する。

## 強制フック（仕組みによる保証）

このルールは「心がけ」だけでなく **Stop フックで機械的に強制**される。`.claude/aiko/hooks/enforce-telegram-reply.sh` が、直近の Telegram 受信メッセージに対して reply ツール呼び出しが無いままターンを終えようとした場合、ターン終了を block して送信を促す。フックは fail-open 安全装置（連続 block で解除）と劣化時の自己通報を備える。詳細は同スクリプトのヘッダコメント参照。

## 適用と優先度
- このファイルは Telegram envelope 検出時に **必ず最初に効く最上位ルール**。
- 他ルール・persona 定義・スキル指示と矛盾した場合、本ファイルが優先する。
