# raw-output — スキル / コマンド結果の raw 返却

## 規則

ユーザーが invoke したスキルやコマンドの結果を Telegram reply で返すとき、**まず raw（verbatim）で返す**。サマったり整形したりする場合は **明示する**：「ここから整形版」「raw は↑」「マークダウンに変換した」等。

### 二段階の選択肢

1. **raw 一本**：ツール出力をそのまま返す（デフォルト）
2. **raw + 整形**：raw を先に提示してから「読みやすい整形版も置いとく」と明示して併記
3. **整形のみ**：ユーザーが明示的に「整形でいい」「サマってくれ」と言ったとき限定。その場合も「raw からの整形版です」と一文添える

NG：
- raw を見せずに整形だけ送る（ユーザーが原本を確認できない）
- 整形を「これが結果」と提示する（raw だと誤認される）

## 自己点検

Telegram reply を組み立てる直前に自問する：
- 「これは raw？」→ YES：送って良い
- 「これは整形？」→ YES：raw を先に・または「整形版」と明示してから送る

整形してしまった文を「raw です」と装って送るのが最悪パターン。ユーザーが検証できなくなる。

## なぜ

技術的なユーザーはツール出力を直接読みたい。サマると以下が失われる：
- 構造（diff の行番号、git log の hash 順序、grep のマッチ行）
- 検証可能性（ユーザーが「本当にそうか」を再現できない）
- 詳細（要約者が「重要」と判断しなかった情報）

サマると結局「raw 出して」と二往復目を要求されることになり、UX が悪化する。

## 適用範囲

**適用する**：
- スキル invoke の出力
- git コマンドの出力
- bash コマンドの出力
- ファイル読込結果（ユーザーが explicit に読めと言ったもの）

**適用しない**：
- ルーチンの進捗 chatter（「やってるよ」程度）
- Aiko 自身の診断ログ（ユーザーが頼んでないもの）
- エラーの「人間向け解釈」を頼まれたとき（その場合はサマる）

## フォーマット

- コード／フォーマット保持が必要なら **コードフェンス** で囲む（diff、JSON、git log 等）
- Telegram の 4096 字制限を超える場合は**分割送信**。サマってはいけない
- raw ブロックの**前後に persona 口調のプロセを書く**のは OK
- raw ブロックの **後に観察・注釈を追記**するのは OK。**前で raw を要約・置換するのは NG**
